# PDF & Documents Tools - Implementation Plan

## Goal

Add two trusted first-party runtime tools to the fixed LemmingsOS tool catalog so Lemmings can create printable documents from files in their instance WorkArea:

```text
file.md   -> documents.markdown_to_html -> file.html
file.html -> documents.print_to_pdf     -> file.pdf
```

```text
file.md   -> documents.print_to_pdf -> internal Markdown to HTML -> file.pdf
file.html -> documents.print_to_pdf -> file.pdf
file.txt  -> documents.print_to_pdf -> simple printable HTML wrapper -> file.pdf
image.png -> documents.print_to_pdf -> simple printable HTML wrapper -> file.pdf
```

This MVP keeps document generation simple:

- The Lemming writes Markdown, HTML, text, CSS, or image files in the instance WorkArea.
- The runtime tools convert or print files inside that WorkArea.
- Generated HTML and PDF files are written back into the WorkArea.
- Artifact persistence and promotion remain separate future capabilities.

## Current Architecture Alignment

The app currently has a fixed runtime catalog in:

```text
lib/lemmings_os/tools/catalog.ex
```

Current tools:

| Tool | Purpose |
|---|---|
| `fs.read_text_file` | Read UTF-8 files from the instance WorkArea |
| `fs.write_text_file` | Write UTF-8 files inside the instance WorkArea |
| `web.search` | Search the web for short snippets |
| `web.fetch` | Fetch HTTP(S) content from one URL |

The new tools must follow the existing runtime shape in `LemmingsOs.Tools.Runtime`:

```elixir
{:ok, %{summary: binary(), preview: binary() | nil, result: map()}}
{:error, %{code: binary(), message: binary(), details: map()}}
```

Do not introduce a second top-level result format such as `%{status: "ok"}` or `%{reason: ...}`.

## New Tools

| Tool | Purpose | Backend |
|---|---|---|
| `documents.markdown_to_html` | Convert an instance WorkArea `.md` file into an instance WorkArea `.html` file | Earmark |
| `documents.print_to_pdf` | Print an instance WorkArea file into an instance WorkArea `.pdf` file | Earmark + Gotenberg |

Implementation target:

- Add a native first-party adapter, likely `LemmingsOs.Tools.Adapters.Documents`.
- Dispatch both tools from `LemmingsOs.Tools.Runtime`.
- Add both tools to `LemmingsOs.Tools.Catalog`.
- Reuse `LemmingsOs.Tools.WorkArea.resolve/2` for all agent-controlled WorkArea paths.
- Use `Req` for all Gotenberg HTTP calls.

Add Earmark as an explicitly approved dependency for this task.

Recommended dependency:

Reason:

- Is adequate for Markdown documents generated by Lemmings.

## Tool 1: `documents.markdown_to_html`

### Purpose

Convert a Markdown file in the instance WorkArea into an HTML file in the same WorkArea.

This tool is useful independently when a Lemming wants an HTML version of a Markdown document without immediately generating a PDF.

### Input

```json
{
  "source_path": "quotes/acme.md",
  "output_path": "quotes/acme.html",
  "overwrite": false
}
```

| Option | Required? | Rules |
|---|---:|---|
| `source_path` | Yes | WorkArea-relative `.md` file |
| `output_path` | Yes | WorkArea-relative `.html` or `.htm` file |
| `overwrite` | No | Default `false`; fail if output exists unless `true` |

### Success Result

Adapter result before runtime normalization:

```elixir
{:ok,
 %{
   summary: "Converted quotes/acme.md to quotes/acme.html",
   preview: "<!doctype html>...",
   result: %{
     "source_path" => "quotes/acme.md",
     "output_path" => "quotes/acme.html",
     "content_type" => "text/html",
     "bytes" => 12840
   }
 }}
```

Runtime result must preserve the existing runtime envelope:

```elixir
{:ok,
 %{
   tool_name: "documents.markdown_to_html",
   args: args,
   summary: "Converted quotes/acme.md to quotes/acme.html",
   preview: "<!doctype html>...",
   result: %{...}
 }}
```

### Markdown Rendering

- Use Earmark to render Markdown into HTML.
- Wrap rendered HTML in a minimal complete HTML document with UTF-8 charset.
- Include a small default printable CSS block for basic readability.
- Do not render EEx, HEEx, Liquid, or any executable template language.
- Do not log Markdown or HTML contents.

## Tool 2: `documents.print_to_pdf`

### Purpose

Print an instance WorkArea file into a PDF file in the same WorkArea.

Supported source types:

| Source type | Default behavior |
|---|---|
| `.html` / `.htm` | Print as HTML |
| `.md` | Convert Markdown to HTML internally, then print PDF |
| `.txt` | Wrap raw text in a simple printable HTML page |
| `.png` / `.jpg` / `.jpeg` / `.webp` | Wrap image in a simple printable HTML page |

### Input

```json
{
  "source_path": "quotes/body.html",
  "header_path": "quotes/body_pdf_header.html",
  "footer_path": "quotes/body_pdf_footer.html",
  "style_paths": ["quotes/body_pdf.css"],
  "output_path": "quotes/acme.pdf",
  "paper_size": "A4",
  "landscape": false,
  "margin_top": "0.5in",
  "margin_bottom": "0.5in",
  "margin_left": "0.5in",
  "margin_right": "0.5in",
  "overwrite": false
}
```

| Option | Required? | Rules |
|---|---:|---|
| `source_path` | Yes | WorkArea-relative supported source file |
| `output_path` | Yes | WorkArea-relative `.pdf` file |
| `header_path` | No | WorkArea-relative `.html` or `.htm` file |
| `footer_path` | No | WorkArea-relative `.html` or `.htm` file |
| `style_paths` | No | List of WorkArea-relative `.css` files |
| `paper_size` | No | Default `A4` |
| `landscape` | No | Default `false` |
| `margin_top` | No | Optional Gotenberg-compatible string |
| `margin_bottom` | No | Optional Gotenberg-compatible string |
| `margin_left` | No | Optional Gotenberg-compatible string |
| `margin_right` | No | Optional Gotenberg-compatible string |
| `print_raw_file` | No | Default `false`; for `.md`, print source as raw text instead of rendered Markdown |
| `overwrite` | No | Default `false`; fail if output exists unless `true` |

### Success Result

Adapter result before runtime normalization:

```elixir
{:ok,
 %{
   summary: "Printed quotes/body.html to quotes/acme.pdf",
   preview: nil,
   result: %{
     "source_path" => "quotes/body.html",
     "output_path" => "quotes/acme.pdf",
     "content_type" => "application/pdf",
     "bytes" => 48291
   }
 }}
```

## Header, Footer, And CSS Resolution

`documents.print_to_pdf` supports explicit WorkArea files, conventional WorkArea siblings, and operator-configured global fallbacks.

Precedence:

1. Explicit tool arguments win.
2. If no explicit path is passed, look for conventional WorkArea sibling files based on the source basename.
3. If no conventional WorkArea file exists, use configured operator fallback files under `priv/documents/` when configured and present.
4. If nothing is found, print without that header, footer, or CSS.

Conventions:

| Source | Header | Footer | CSS |
|---|---|---|---|
| `report.html` | `report_pdf_header.html` | `report_pdf_footer.html` | `report_pdf.css` |
| `reports/q1.md` | `reports/q1_pdf_header.html` | `reports/q1_pdf_footer.html` | `reports/q1_pdf.css` |
| `notes.txt` | `notes_pdf_header.html` | `notes_pdf_footer.html` | `notes_pdf.css` |

Operator fallback env vars:

```text
LEMMINGS_DOCUMENTS_DEFAULT_HEADER_PATH=priv/documents/default_header.html
LEMMINGS_DOCUMENTS_DEFAULT_FOOTER_PATH=priv/documents/default_footer.html
LEMMINGS_DOCUMENTS_DEFAULT_CSS_PATH=priv/documents/default.css
```

Fallback env files are operator-controlled deployment assets:

- They are never passed by agents.
- They are not resolved through `WorkArea.resolve/2`.
- For MVP, they must live under `priv/documents/`.
- They must be readable regular files.
- Symlinks are rejected.
- Header and footer fallbacks must use `.html` or `.htm`.
- CSS fallbacks must use `.css`.
- They must pass fallback asset size checks.
- Missing fallback files are ignored.
- Invalid fallback extensions are ignored and logged safely.
- Fallback file paths and file contents must not be copied into tool results or logs.

Explicit `header_path`, `footer_path`, and `style_paths` are agent-controlled WorkArea-relative paths and must always be resolved through `WorkArea.resolve/2`.

## PDF Backend

Use Gotenberg for HTML-to-PDF rendering.

Recommended image:

```text
gotenberg/gotenberg:8
```

Reason:

- Docker-friendly.
- Provides an HTTP API for PDF conversion.
- Avoids installing Chromium directly in the Phoenix app container.
- Supports HTML printing with header, footer, CSS, margins, paper size, and orientation options.

### Gotenberg Request Contract

Use `Req` to send multipart requests to the configured Gotenberg URL.

Target endpoint:

```text
POST {LEMMINGS_GOTENBERG_URL}/forms/chromium/convert/html
```

Multipart fields:

- `files`: generated or source HTML body as `index.html`.
- `files`: optional header HTML as `header.html`.
- `files`: optional footer HTML as `footer.html`.
- `files`: optional CSS files with stable generated names such as `style-1.css`, `style-2.css`.
- Form fields for Gotenberg options: paper size, orientation, margins, and related Chromium conversion options.

CSS files from `style_paths`, conventions, and fallbacks are injected or linked by the runtime into the generated print HTML. Source HTML does not need to reference generated multipart names such as `style-1.css`.

Rules:

- The agent never supplies the Gotenberg URL.
- Only the runtime builds the HTTP request.
- Backend response body is written to a temporary file in the target directory only after status validation passes.
- The temporary file is size-validated and then atomically renamed to `output_path`.
- Failed conversion or failed size validation must not leave a partial PDF at `output_path`.
- Non-2xx responses return a safe `tool.documents.pdf_conversion_failed` error.
- Connection failures and timeouts return `tool.documents.pdf_backend_unavailable`.
- Validation errors, path errors, missing files, unsupported formats, and output conflicts are never retried.
- Transient backend failures may be retried according to config.

## Configuration

Add runtime configuration in `config/runtime.exs`, following the existing project pattern.

Environment variables:

```text
LEMMINGS_GOTENBERG_URL=http://gotenberg:3000
LEMMINGS_DOCUMENTS_PDF_TIMEOUT_MS=30000
LEMMINGS_DOCUMENTS_PDF_CONNECT_TIMEOUT_MS=5000
LEMMINGS_DOCUMENTS_PDF_RETRIES=1
LEMMINGS_DOCUMENTS_MAX_SOURCE_BYTES=10485760
LEMMINGS_DOCUMENTS_MAX_PDF_BYTES=52428800
LEMMINGS_DOCUMENTS_MAX_FALLBACK_BYTES=1048576
LEMMINGS_DOCUMENTS_DEFAULT_HEADER_PATH=
LEMMINGS_DOCUMENTS_DEFAULT_FOOTER_PATH=
LEMMINGS_DOCUMENTS_DEFAULT_CSS_PATH=
```

Defaults:

| Env var | Default | Notes |
|---|---:|---|
| `LEMMINGS_GOTENBERG_URL` | `http://gotenberg:3000` | Compose/release default internal service URL |
| `LEMMINGS_DOCUMENTS_PDF_TIMEOUT_MS` | `30000` | Full conversion timeout |
| `LEMMINGS_DOCUMENTS_PDF_CONNECT_TIMEOUT_MS` | `5000` | Backend connect timeout |
| `LEMMINGS_DOCUMENTS_PDF_RETRIES` | `1` | Retry transient backend failures only |
| `LEMMINGS_DOCUMENTS_MAX_SOURCE_BYTES` | `10485760` | 10 MB input limit |
| `LEMMINGS_DOCUMENTS_MAX_PDF_BYTES` | `52428800` | 50 MB generated PDF limit |
| `LEMMINGS_DOCUMENTS_MAX_FALLBACK_BYTES` | `1048576` | 1 MB per operator fallback asset |
| `LEMMINGS_DOCUMENTS_DEFAULT_HEADER_PATH` | unset | Operator fallback header |
| `LEMMINGS_DOCUMENTS_DEFAULT_FOOTER_PATH` | unset | Operator fallback footer |
| `LEMMINGS_DOCUMENTS_DEFAULT_CSS_PATH` | unset | Operator fallback CSS |

These env vars are deployment/runtime adapter configuration. They are not hierarchy policy, not inherited Lemming configuration, and not controllable by the LLM.

Local dev and non-Compose deployments may override `LEMMINGS_GOTENBERG_URL` to point at an externally managed Gotenberg service.

Store parsed values under a documents-specific application config, for example:

```elixir
config :lemmings_os, :documents,
  gotenberg_url: "...",
  pdf_timeout_ms: 30_000,
  pdf_connect_timeout_ms: 5_000,
  pdf_retries: 1,
  max_source_bytes: 10 * 1024 * 1024,
  max_pdf_bytes: 50 * 1024 * 1024,
  max_fallback_bytes: 1 * 1024 * 1024,
  default_header_path: nil,
  default_footer_path: nil,
  default_css_path: nil
```

Invalid numeric env values should fail startup with a clear error rather than silently changing safety limits.

## Docker Compose

The user-facing Docker Compose setup should run Gotenberg on the same private Compose network as the Lemmings app services. Gotenberg must be reachable by the app containers and not published to the host by default.

Add a Gotenberg service on an internal app network:

```yaml
gotenberg:
  image: gotenberg/gotenberg:8
  expose:
    - "3000"
  networks:
    - lemmings_internal
```

The Phoenix/runtime service should call:

```text
http://gotenberg:3000
```

Compose rules:

- Do not publish Gotenberg with `ports` in the default user-facing compose file.
- Do not run Gotenberg with `network_mode: host`.
- Put the app services that execute tools on the same private network as Gotenberg.
- Add comments showing developers may point `LEMMINGS_GOTENBERG_URL` at an external local Gotenberg for dev-only workflows.
- Operators must not expose Gotenberg publicly.

## WorkArea Safety Rules

All agent-controlled paths must be WorkArea-relative. Lemmings pass WorkArea-relative paths such as `quotes/acme.html`; the runtime resolves them through `WorkArea.resolve/2`.

Allowed examples:

```text
quotes/acme.md
quotes/acme.html
quotes/acme.pdf
contracts/body.html
contracts/styles.css
images/logo.png
notes/raw.txt
```

Rejected examples:

```text
/etc/passwd
/tmp/file.html
../secret.env
~/.ssh/id_rsa
C:\Users\user\file.html
```

Rules:

- Reject absolute paths.
- Reject path traversal.
- Reject Windows drive paths and backslashes, consistent with `WorkArea.resolve/2`.
- Reject paths outside the instance WorkArea.
- Reject symlink traversal using the existing WorkArea behavior.
- Reject unsupported file extensions.
- Do not log full host paths.
- Do not log document contents.
- Do not expose absolute WorkArea paths in tool results.

Generated output files must be written atomically:

- Validate the source, options, output path, overwrite policy, and expected output extension before writing.
- Write to a temporary file in the same target directory.
- Validate generated file size before finalization.
- Rename the temp file into `output_path` only after validation succeeds.
- Failed conversion, failed write, or failed size validation must not leave a partial final file at `output_path`.
- Clean up temporary files best-effort on failure.

## HTML And Asset Policy

MVP supports:

- WorkArea source HTML.
- Runtime-generated HTML for Markdown, raw text, and image wrappers.
- Explicit WorkArea header/footer/CSS files.
- Conventional WorkArea sibling header/footer/CSS files.
- Operator-configured fallback header/footer/CSS files.

MVP explicitly blocks:

- Remote HTML assets referenced with `http://`, `https://`, or protocol-relative URLs like `//example.com`.
- Local file references such as `file://`.
- CSS `@import`.
- Remote images, fonts, stylesheets, scripts, frames, media, or links intended to be fetched by Chromium.

Reject source HTML, generated Markdown HTML, generated wrappers, header/footer HTML, and CSS when obvious blocked references are present. This is a conservative string/pattern guard for MVP, not a complete HTML/CSS security parser. Fail safely before calling Gotenberg. Remote asset support requires a future explicit tool policy and network allowlist.

MVP also does not support template rendering.

Image support is a basic single-image printable wrapper only. Do not promise layout guarantees, pagination controls, image resizing options, or a multi-image layout engine.

## Error Behavior

Use structured, safe, namespaced errors.

| Case | Error code |
|---|---|
| Invalid args | `tool.validation.invalid_args` |
| Invalid WorkArea path | `tool.validation.invalid_path` |
| Source file missing | `tool.documents.source_not_found` |
| Header/footer/style file missing when explicitly requested | `tool.documents.asset_not_found` |
| Unsupported extension | `tool.documents.unsupported_format` |
| Output exists and overwrite is false | `tool.documents.output_exists` |
| Source file too large | `tool.documents.file_too_large` |
| Gotenberg unavailable or timed out | `tool.documents.pdf_backend_unavailable` |
| Gotenberg conversion failed | `tool.documents.pdf_conversion_failed` |
| Generated PDF exceeds limit | `tool.documents.pdf_too_large` |
| Output write failed | `tool.documents.write_failed` |

Example:

```elixir
{:error,
 %{
   code: "tool.documents.source_not_found",
   message: "Source file not found",
   details: %{"source_path" => "quotes/acme.md"}
 }}
```

Details must use WorkArea-relative paths only.

## Observability

Emit safe structured logs or telemetry for:

| Event | Default level | Include |
|---|---|---|
| Markdown conversion started | `debug` | WorkArea-relative source path, WorkArea-relative output path |
| Markdown conversion completed | `info` | WorkArea-relative output path, byte size |
| Markdown conversion failed | `warning` | reason code |
| PDF print started | `debug` | WorkArea-relative source path, WorkArea-relative output path, source type |
| PDF print completed | `info` | WorkArea-relative output path, byte size, duration |
| PDF backend unavailable | `warning` | reason code, backend name |
| PDF conversion failed | `warning` | reason code, backend name, status if available |
| Unexpected tool error | `error` | reason code only |

Do not include:

- document contents
- generated HTML
- absolute host paths
- raw WorkArea root
- fallback env file paths
- secrets
- raw backend response bodies
- full exception dumps
